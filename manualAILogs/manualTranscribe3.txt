This is the minimal ASP.NET Core startup flow in Program.cs.

## How it works

1. `var builder = WebApplication.CreateBuilder(args);`
   - Creates the host builder and application configuration.
   - Sets up services, configuration, logging, and environment.

2. `builder.Services.AddControllersWithViews();`
   - Registers MVC controller + view support in dependency injection.
   - This enables controller classes to be discovered and Razor views to be rendered.

3. `var app = builder.Build();`
   - Builds the `WebApplication` instance.
   - At this point, the DI container is finalized and middleware can be configured.

4. Middleware pipeline configuration:
   - `app.UseExceptionHandler("/Home/Error");`
     - In non-development mode, catches exceptions and shows a friendly error page.
   - `app.UseHsts();`
     - Adds HTTP Strict Transport Security for browsers when using HTTPS.
   - `app.UseHttpsRedirection();`
     - Redirects HTTP requests to HTTPS.
   - `app.UseStaticFiles();`
     - Serves files from wwwroot like CSS, JS, images.
   - `app.UseRouting();`
     - Enables endpoint routing.
   - `app.UseAuthorization();`
     - Adds authorization checks before request execution.

5. `app.MapControllerRoute(...)`
   - Defines the default route pattern:
     - `{controller=Home}` means use `HomeController` by default
     - `{action=Index}` means use `Index` action by default
     - `{id?}` means optional `id` parameter
   - This is what maps incoming URLs to controllers and actions.

6. `app.Run();`
   - Starts the web server and begins processing incoming requests.

## Why it works

- ASP.NET Core uses dependency injection + middleware to build the request pipeline.
- `CreateBuilder` sets up the host and services automatically, which is the new minimal hosting model in .NET 6+.
- The order of `Use...` calls matters:
  - static files must be served before routing/mvc
  - routing must be enabled before mapping controller routes
- `MapControllerRoute` connects the route pattern to the controllers registered by `AddControllersWithViews`.

So this file is the app startup: configure services, compose middleware, define routing, then run the app.

Can you explain to me the following terms: Razor views, DI container and middleware

## Razor views

- Razor views are `.cshtml` files that mix HTML with C# code.
- They are the UI templates used by MVC controllers to generate HTML responses.
- Example: Index.cshtml
- Razor syntax uses `@` for C# expressions and control flow, e.g. `@Model.Title`.

Why it matters:
- lets you render dynamic data from your controller model into HTML.
- keeps presentation separate from controller logic.

## DI container

- DI means Dependency Injection.
- The DI container is the built-in object that creates and supplies service instances.
- In Program.cs, `builder.Services.AddControllersWithViews()` registers services into that container.
- Then controllers and other classes can request dependencies via constructor parameters.

Why it matters:
- it makes code more modular, testable, and easier to change.
- the container manages lifetime and resolves dependencies automatically.

## Middleware

- Middleware are components that process each HTTP request in order.
- They sit in a pipeline, one after another.
- Examples in your app:
  - `UseHttpsRedirection()`
  - `UseStaticFiles()`
  - `UseRouting()`
  - `UseAuthorization()`
- Each middleware can inspect/modify the request, call the next component, and/or generate a response.

Why it matters:
- middleware defines how requests are handled before they reach controllers.
- order matters: static files should be served before routing, authorization should happen after routing is established, etc.

In short:
- Razor views = HTML + C# templates for responses
- DI container = the service factory for reusable dependencies
- Middleware = the ordered request-processing pipeline

What types of connetctions (like 1:1,N:1) do i have and what are they?

In your model the relationships are expressed as object references and lists.

Many-to-many (N:M)
Player ↔ Team

Team.Players is a list of players
Player.Teams is a list of teams
This is true many-to-many: one player can belong to many teams, and one team can contain many players.
Team ↔ Tournament

Tournament.Teams is a list of teams
Team.Tournaments is a list of tournaments
Many teams can enter many tournaments.
One-to-many / many-to-one
Venue → Tournament

Venue.Tournaments is a list
Tournament.Venue is a single object
So one venue hosts many tournaments, each tournament has one venue.
This is 1:N from Venue to Tournament, and N:1 from Tournament to Venue.
Tournament → Match

Tournament.Matches is a list
Match.Tournament is a single object
One tournament has many matches, each match belongs to one tournament.
BoardGame → Match

Each Match.Game is one BoardGame
The reverse is not explicitly stored, but a board game can appear in many matches.
So conceptually 1:N from BoardGame to Match.
Single-reference relationships
Match.TeamA and Match.TeamB
Each match references exactly two teams
Each of those teams can appear in many matches
So each Match has two N:1 relationships to Team.
Summary
N:M: Player–Team, Team–Tournament
1:N: Venue–Tournament, Tournament–Match
N:1: Match → Tournament, Match → BoardGame, Match → TeamA / TeamB
That is how the connection types appear in your current domain model.
